## 1. Résumé Exécutif

L'application **fonctionne au quotidien** : 608 tests passent, la BD locale est peuplée, le draft coach et les analyses tournent. Mais l'écart entre **l'état documenté** et **l'état réel** est devenu majeur, et le projet a un problème structurel unique qui domine tout le reste :

> **Le pipeline de données (scraping LoLalytics) est en crise depuis ~6 mois et a absorbé l'essentiel de l'effort de développement, tout en se dégradant.**

Constats principaux, par ordre de gravité :

1. 🔴 **La BD a perdu 60% de ses données** : 40 753 matchups + 30 668 synergies le 19/03/2026 → **16 179 matchups + 12 943 synergies** aujourd'hui. La granularité multi-lane a disparu au passage.
2. 🔴 **Toute l'automatisation est à l'arrêt** : auto-update local arrêté depuis le 19/03/2026 ; le workflow GitHub Actions de scraping quotidien est **cancelled au timeout de 2h tous les jours** (il brûle ~60h de minutes Actions/mois pour rien) ; la BD Neon servie au .exe distant date du 19/03/2026.
3. 🔴 **La CI est rouge sur master depuis le 24/05/2026** (pylint en échec + 2 tests en échec), et le job de build .exe **ne s'est jamais exécuté** (il cible les branches `main`/`inspiring-rhodes` alors que le repo utilise `master`).
4. 🟠 **La victoire « dette technique » du Sprint 1 a été défaite** : `assistant.py` est repassé de 190 à 1 843 lignes, `lol_coach_legacy.py` (2 356 lignes, « temporaire » depuis décembre 2025) est toujours là. 8 fichiers violent la règle des 500 lignes affichée dans CLAUDE.md.
5. 🟠 **La documentation est massivement désynchronisée** : 3 numéros de version différents, TODO.md auto-contradictoire, README décrivant des fichiers inexistants, ~la moitié des docs obsolètes.

Le précédent audit donnait **A- (18/20) « Production-Ready »**. Cette note ne reflète plus la réalité. Évaluation actuelle : **C+ — application utilisable, infrastructure et processus en dérive**. Le détail et la voie de sortie sont dans `ROADMAP_2026.md`.

## 3. Le Problème Central : le Pipeline de Données

### 3.1 Chronologie de la crise Cloudflare (reconstituée depuis git)

- **Déc. 2025 – Jan. 2026** : scraping parallèle Selenium opérationnel (~12 min, 40k matchups multi-lane). Auto-update local quotidien fonctionnel.
- **Jan.–Fév. 2026** : LoLalytics change son DOM + déploie des protections Cloudflare. Début d'une longue série de correctifs : `cloudflare_detector.py`, retries, délais aléatoires, locales non-anglaises…
- **Fév. 2026** : pivot architecture — API FastAPI sur Render abandonnée (timeouts 60s+), remplacée par connexion **PostgreSQL Neon directe** dans le .exe (CHANGELOG 1.2.0). Scraping déplacé vers GitHub Actions (`scraping.yml`).
- **Mars 2026** : dernier auto-update local réussi le **19/03** (33,8 min, 40 753 matchups synchronisés vers Neon). Puis plus rien.
- **Mai 2026** : tentative de migration **Playwright + stealth** (Tâche #18, commit du 24/05)… puis **revert complet vers Selenium** le 01/06 (`a032b64`). La branche `feature/playwright-migration` reste orpheline.
- **Juin 2026** : re-scrape manuel via `fill_db.py` le 01/06 : 172/172 champions OK, mais **16 179 matchups seulement** (lane par défaut uniquement). Scripts `repair_matchups.py` / `repair_synergies.py` exécutés à la main (dernier run : aujourd'hui 11/06).

### 3.2 État des 3 canaux de mise à jour

| Canal | État | Preuve |
|---|---|---|
| Auto-update local (Task Scheduler) | 🔴 Arrêté | `logs/auto_update.log` : dernière entrée 19/03/2026 |
| GitHub Actions quotidien (`scraping.yml`) | 🔴 Échoue **tous les jours** : cancelled à 2h00 de timeout | `gh run list` : 100% cancelled (12/05 → 16/05 vérifiés, pattern quotidien). Probablement Cloudflare bloque les IP datacenter |
| Manuel (`fill_db.py` + repair) | 🟡 Fonctionne mais incomplet | Log du 01/06. **Ne recalcule pas** `champion_scores` ni `pool_ban_recommendations`, **ne synchronise pas Neon** |

### 3.3 Conséquences concrètes

1. **Perte de granularité lane** : les 40k matchups du printemps incluaient les données multi-lane (cf. migrations server « remove unique constraints for multi-lane », agrégation pondérée dans `get_matchup_delta2`). Le scrape actuel ne couvre que la lane par défaut → ~94 matchups/champion au lieu de ~237.
2. **Le .exe distant sert des données du 19 mars** : le mode `postgresql_only` (auto-activé quand compilé) lit Neon, qui n'a pas été resynchronisé depuis le dernier auto-update. Le mode « gaming café » est donc silencieusement périmé de 3 mois.
3. **`champion_scores` (tier lists) et `pool_ban_recommendations` datent d'avant le re-scrape** : ils ont été calculés sur l'ancienne BD ; le flux manuel actuel ne les rafraîchit pas → tier lists et bans potentiellement incohérents avec les matchups affichés.
4. **Coût caché** : ~2h de runner GitHub Actions consommées **chaque nuit** pour un job qui n'aboutit jamais.

### 3.4 Fragilité structurelle

- XPath **absolus** dans `config_constants.py` (`/html/body/main/div[5]/...`) : toute modification du DOM LoLalytics casse le parsing.
- `fill_db.py` : `PATCH = "14"` **hardcodé** (en violation de la règle config_constants), pas de recalcul des scores, pas de sync Neon.
- Le projet dépend à 100% d'un site tiers protégé par Cloudflare, sans aucune source de données alternative ni plan B documenté.

### 4.2 Duplication client/serveur

`server/src/analysis/scoring.py` est un **copier-coller octet pour octet** de `src/analysis/scoring.py` (16 660 octets identiques), idem `models.py` (12 123 octets), et `team_analysis.py`, `recommendations.py`, `tier_list.py` sont quasi identiques. Toute correction d'algorithme doit être faite deux fois — et rien ne garantit que c'est le cas (les fichiers ont déjà divergé : `tier_list.py` 12 170 vs 12 274 octets).

### 4.3 Couche données : 4 implémentations + 1 zombie

`DataSource` est implémenté par : `SQLiteDataSource`, `PostgreSQLDataSource`, `HybridDataSource`, et `APIDataSource` (zombie post-1.2.0, avec `BASE_URL` pointant encore vers Render). Trois modes configurables (`sqlite_only` / `postgresql_only` / `hybrid`) auto-commutés selon que l'app est compilée ou non. **C'est trop de surface pour un projet mono-utilisateur** — chaque mode multiplie les chemins de code, les tests et les bugs possibles (cf. `get_reverse_matchups_for_draft` manquant dans Hybrid, corrigé en jan. 2026).

### 4.4 Points forts (à préserver)

- ✅ Requêtes SQL **toutes paramétrées** (vérifié par sondage + bandit en CI).
- ✅ Dataclasses immutables (`models.py`) avec validation — migration réussie.
- ✅ `config_constants.py` riche et bien documenté (à l'exception des XPath absolus et de quelques valeurs résiduelles hardcodées).
- ✅ Culture du **test de régression par bug** réellement appliquée (dossier `tests/regression/`, 40+ fichiers de tests, 608 tests qui passent).
- ✅ Alembic en place et à jour (`cc46f5edf9b2`), côté client et serveur.
- ✅ Cache bidirectionnel + bulk loading : le problème N+1 (1h06 → 20 s) a été résolu proprement.
